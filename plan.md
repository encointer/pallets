# Private Offline Payments ‚Äî Design Plan

## Motivation

The current offline-payment pallet uses ZK proofs but reveals the sender on-chain (the sender's `AccountId` is an explicit parameter, looked up in `OfflineIdentities`). This makes ZK unnecessary ‚Äî a simple signature achieves the same. To justify ZK, we need **sender anonymity**: hide *which* registered identity is paying, within an anonymity set of all depositors.

## Architecture: Shielded Note Pool

Move from account-based debits to a UTXO/note model. Users deposit funds into a shared shielded pool. Payments spend notes and create new ones without revealing which note was spent.

### Note Structure

```
note = (value, secret, blinding)
note_commitment = Poseidon(value, secret, blinding)
nullifier = Poseidon(secret, leaf_index)
```

- `value`: amount in **reference units** (see Demurrage section ‚Äî not raw CC amounts)
- `secret`: per-note random secret (Fr), generated by the note creator
- `blinding`: random blinding factor (Fr) for commitment uniqueness
- `leaf_index`: position in the on-chain Merkle tree (prevents nullifier linkability across notes)

### On-Chain Storage

1. **Incremental Merkle Tree** (new storage)
   - Append-only tree of `note_commitment` values
   - Fixed depth = 20 (supports ~1M notes, sufficient for community currencies)
   - Poseidon hash for internal nodes (SNARK-friendly, already in the codebase)
   - Store: `leaves: Vec<[u8;32]>`, `next_index: u32`, precomputed subtree hashes for efficient appends
   - Alternatively, store only the frontier (one hash per level) for O(depth) append cost

2. **Recent Merkle Roots** (new storage)
   - Ring buffer of last N roots (e.g., N=100)
   - Allows proofs built against slightly stale state (critical for offline use ‚Äî sender caches their Merkle path before going offline)
   - On each new leaf insertion, push the new root

3. **Used Nullifiers** (existing, reused as-is)
   - `Map<[u8; 32] ‚Üí ()>` ‚Äî shared across all payment types

4. **Demurrage Epoch Log** (new storage, per community)
   - Append-only list of `(block_number, cumulative_demurrage_factor)` entries
   - One entry per demurrage rate change (see Demurrage section)

5. **Remove**: `OfflineIdentities` map ‚Äî no longer needed (no account‚Üîcommitment link)

### Demurrage Handling

Encointer community currencies apply demurrage: `balance *= e^(-rate √ó blocks_elapsed)`. A shielded note that locks in a fixed value would let users bypass demurrage entirely ‚Äî defeating the purpose of community currencies.

#### Core Idea: Reference-Value Normalization

All note values are stored in **epoch-0 normalized reference units**, not raw CC amounts. Demurrage is applied only at the pool boundaries (deposit and withdraw), never during private transfers.

#### Cumulative Demurrage Factor

The chain maintains a **cumulative demurrage factor** `F(b)` for each community ‚Äî the total decay from block 0 to block `b`. This accounts for rate changes over time:

```
F(b) = e^(-r‚ÇÄ √ó b‚ÇÅ) √ó e^(-r‚ÇÅ √ó (b‚ÇÇ - b‚ÇÅ)) √ó ... √ó e^(-r‚Çñ √ó (b - b‚Çñ))
```

where `r‚ÇÄ, r‚ÇÅ, ..., r‚Çñ` are the demurrage rates and `b‚ÇÅ, b‚ÇÇ, ..., b‚Çñ` are the blocks at which rate changes occurred.

Rather than recomputing this product every time, the chain stores an **epoch log**: a list of `(block_number, cumulative_factor)` snapshots taken at each rate change. Between rate changes, the factor is extended analytically:

```
F(b) = F(b‚Çñ) √ó e^(-r‚Çñ √ó (b - b‚Çñ))
```

where `b‚Çñ` is the most recent rate-change block and `r‚Çñ` is the current rate.

#### On-Chain Storage: `DemurrageEpochLog`

```rust
/// One entry per demurrage rate change for a community.
#[derive(Encode, Decode, TypeInfo, Clone)]
pub struct DemurrageEpoch<BlockNumber> {
    /// Block at which this rate took effect
    pub block: BlockNumber,
    /// Cumulative demurrage factor at `block` (BalanceType, in [0, 1])
    /// F(block) = product of all prior decay
    pub cumulative_factor: BalanceType,
    /// The demurrage rate per block that applies from `block` onward
    pub rate: Demurrage,
}

/// Per-community append-only log. Initialized with one entry at genesis/community creation.
DemurrageEpochLog: StorageMap<CommunityIdentifier, BoundedVec<DemurrageEpoch, MaxEpochs>>
```

`MaxEpochs` can be generous (e.g., 256) since rate changes are rare (months to years apart).

**Initialization**: When a community is created (or the shielded pool is first used), insert:
```
DemurrageEpoch { block: 0, cumulative_factor: 1.0, rate: initial_demurrage_rate }
```

**On rate change**: When governance calls `update_demurrage()` at block `b_new` with new rate `r_new`:
1. Read the last epoch entry `(b‚Çñ, F‚Çñ, r‚Çñ)`
2. Compute `F(b_new) = F‚Çñ √ó e^(-r‚Çñ √ó (b_new - b‚Çñ))`
3. Append `DemurrageEpoch { block: b_new, cumulative_factor: F(b_new), rate: r_new }`

#### At Deposit (block `d`)

```
F_now = F(d)    // computed from last epoch entry
reference_value = actual_value / F_now
```

This "un-decays" the value to epoch-0 units. The note commitment uses `reference_value`.

#### At Withdrawal (block `w`)

```
F_now = F(w)
payout = reference_value √ó F_now
```

Correctly applies all cumulative demurrage from epoch 0 through block `w`, including any rate changes that happened while the note was in the pool.

#### During Private Transfer

The circuit enforces `ref_val_in = ref_val_out_1 + ref_val_out_2` ‚Äî pure value conservation in reference units. **No demurrage computation in the circuit.** Both input and output notes are in the same normalized space.

When the sender pays `actual_amount` at block `t`, the wallet computes:
```
F_t = F(t)      // from cached epoch log + current rate
ref_val_recipient = actual_amount / F_t
ref_val_change = ref_val_in - ref_val_recipient
```

If the recipient withdraws immediately at block `t`: `payout = ref_val_recipient √ó F_t = actual_amount`. Correct.
If the recipient waits until block `t + Œî`: `payout = actual_amount √ó F(t+Œî)/F(t) = actual_amount √ó e^(-r √ó Œî)`. Also correct ‚Äî the note demurges from the moment of payment.

#### Offline Accuracy

The sender needs `F(t)` to compute reference values. They can cache the epoch log (tiny ‚Äî one entry per rate change) and the current rate when last online. From there:

```
F(t) ‚âà F(last_sync) √ó e^(-r_current √ó (t - last_sync))
```

Accuracy depends on two things:
1. **Block estimate error** ‚Äî sender estimates `t` from wall-clock time. At rate ~1.1e-7/block, being 1000 blocks (~1.7 hours) off introduces error ~0.01%. Well under the 0.1% threshold.
2. **Rate change while offline** ‚Äî if governance changes the rate while the sender is offline, the sender uses the old rate. For a rate change from r‚ÇÅ to r‚ÇÇ at block b_change, the error over Œî blocks after the change is `|e^(-r‚ÇÅ√óŒî) - e^(-r‚ÇÇ√óŒî)|`. For typical rate changes (e.g., halving time moves from 1 year to 11 months), this is negligible over the ~10 minute offline window.

**Worst case analysis**: Sender is offline for 10 minutes (100 blocks). Demurrage rate doubles (extreme scenario). Error = `|e^(-r√ó100) - e^(-2r√ó100)|` ‚âà `r √ó 100` ‚âà 0.001%. Far below the 0.1% tolerance.

#### Privacy Impact

The epoch log is public and global per community ‚Äî it reveals nothing about individual notes or payments. The reference-value normalization is computed off-chain by wallets. On-chain, only commitments (opaque hashes) are stored. No timing information leaks from note values.

#### Pool Solvency

The pool holds actual CC tokens in a `BalanceEntry` that demurges via the standard mechanism. The invariant:

```
pool_balance = Œ£ (reference_value_i √ó F(current_block))   for all unspent notes i
```

This holds because:
- At deposit: pool receives `actual_value`, note stores `actual_value / F(d)`. Pool balance at block `b > d`: `actual_value √ó F(b)/F(d)`. Note's withdrawal value at `b`: `(actual_value / F(d)) √ó F(b)` = same.
- Private transfers: pool balance unchanged, reference values conserved.
- At withdrawal: pool pays `reference_value √ó F(w)`, which matches what the pool has decayed to.

Small rounding errors from fixed-point arithmetic accumulate over many transactions but remain negligible (< 1 ULP per operation on i64F64).

### New Circuit: `PrivatePaymentCircuit`

**Public inputs (6 field elements):**

| # | Field | Purpose |
|---|-------|---------|
| 1 | `merkle_root` | Which tree state the proof is against |
| 2 | `nullifier` | Double-spend prevention |
| 3 | `out_commitment_1` | Recipient's new note commitment |
| 4 | `out_commitment_2` | Sender's change note commitment (or zero-value note) |
| 5 | `asset_hash` | Community + chain binding (`blake2(cid ‚à• genesis_hash)`) |
| 6 | `fee_or_metadata` | Reserved (e.g., for relayer fees or future extensions) |

**Private inputs:**

| Field | Purpose |
|-------|---------|
| `in_value` | Reference-value of the spent note |
| `in_secret` | Secret of the spent note |
| `in_blinding` | Blinding of the spent note |
| `in_leaf_index` | Position of the spent note in the tree |
| `merkle_path[20]` | Sibling hashes along the path to root |
| `out_value_1` | Reference-value going to recipient |
| `out_secret_1` | Recipient's note secret |
| `out_blinding_1` | Recipient's note blinding |
| `out_value_2` | Reference-value as change back to sender |
| `out_secret_2` | Sender's change note secret |
| `out_blinding_2` | Sender's change note blinding |

**R1CS constraints:**

1. **Input note reconstruction**: `in_commitment = Poseidon(in_value, in_secret, in_blinding)`
2. **Merkle membership**: Verify `in_commitment` at `in_leaf_index` with `merkle_path` hashes up to `merkle_root` (20 Poseidon hashes)
3. **Nullifier correctness**: `nullifier = Poseidon(in_secret, in_leaf_index)`
4. **Output note 1**: `out_commitment_1 = Poseidon(out_value_1, out_secret_1, out_blinding_1)`
5. **Output note 2**: `out_commitment_2 = Poseidon(out_value_2, out_secret_2, out_blinding_2)`
6. **Value conservation**: `in_value = out_value_1 + out_value_2`
7. **Non-negative outputs**: Range checks on `out_value_1` and `out_value_2` (e.g., check fits in 64 bits ‚Äî ~128 constraints each using binary decomposition)

**Estimated constraint count**: ~6,000‚Äì8,000 (very manageable for Groth16; proof generation ~2-5s on a phone).

**Note**: All values in the circuit are in reference units. Demurrage is invisible to the circuit ‚Äî it only enforces conservation.

### Extrinsics

#### 1. `deposit(amount, cid, note_commitment)` ‚Äî Shield funds

- Caller pays `amount` (actual CC) from their balance (public, no ZK needed)
- Chain computes `F(current_block)` from the epoch log
- Chain debits caller, transfers to shielded pool account
- Appends `note_commitment` to Merkle tree
- Emits `Deposited { who, cid, amount, note_commitment, leaf_index }`
- The caller privately records `(reference_value, secret, blinding, leaf_index)` ‚Äî needed to spend later
- **Important**: the chain does NOT verify that the commitment encodes `amount / F(now)`. It can't ‚Äî the commitment is opaque. The depositor must use the correct reference_value or their note will be worth the wrong amount at withdrawal. This is a self-enforcing property: incorrect reference_value only hurts the depositor.

#### 2. `submit_private_payment(proof, merkle_root, nullifier, out_commitment_1, out_commitment_2, asset_hash)` ‚Äî Settle a private transfer

- Callable by **anyone** (relayer, recipient, sender when back online)
- No sender identity in the call at all
- Steps:
  1. Check `merkle_root` is in the recent-roots ring buffer
  2. Check `nullifier` not already used
  3. Verify Groth16 proof against public inputs
  4. Append `out_commitment_1` and `out_commitment_2` to the Merkle tree
  5. Mark nullifier as used
  6. Emit `PrivatePaymentSettled { nullifier, out_commitment_1, out_commitment_2 }`

- **Note**: No balance check possible here (we don't know who the sender is!). Value conservation is enforced entirely by the circuit. No demurrage logic ‚Äî all values are in reference units.

#### 3. `withdraw(proof, merkle_root, nullifier, recipient, reference_amount, asset_hash)` ‚Äî Unshield funds

- Proves ownership of a note and reveals the reference_value
- Uses a simpler circuit variant (Merkle membership + nullifier, value is public)
- Chain computes `payout = reference_amount √ó F(current_block)` using the epoch log
- Credits `recipient` with `payout` actual CC from the shielded pool
- The shielded pool is a module-controlled account (PalletId-based pot)

#### 4. `set_verification_key_private(vk)` ‚Äî Governance sets the VK for the new circuit

- Separate VK from the legacy circuit (different constraint system)
- Could store both VKs if backward compatibility needed during migration

This fits comfortably in a single QR code (QR Version 15 at medium ECC handles ~520 bytes of binary data).

### Encrypted Note for Recipient

The recipient needs `(ref_value, secret, blinding)` to later spend their note, plus `(actual_amount, block_estimate)` for UX. Encryption scheme:

1. Sender generates ephemeral x25519 keypair `(eph_sk, eph_pk)`
2. Derives shared secret via ECDH: `shared = x25519(eph_sk, recipient_x25519_pk)`
3. Derives encryption key: `enc_key = blake2(shared ‚à• eph_pk ‚à• recipient_pk)`
4. Encrypts `(ref_value, secret, blinding, actual_amount, block_estimate)` with ChaCha20-Poly1305
5. Includes `eph_pk` in the QR so recipient can reconstruct `shared`

**How recipient gets their x25519 key**: Encointer accounts use sr25519. The recipient's x25519 public key can be derived from their sr25519 key (same underlying curve25519 point), or displayed as a separate QR / NFC tag at point-of-sale.

**Two-round in-person flow**:
1. Recipient displays their receiving key (QR at point of sale, NFC tag, etc.)
2. Sender scans it, constructs proof offline, displays payment QR
3. Recipient scans payment QR ‚Üí submits on-chain when connectivity is available

### Offline Merkle Path Caching

The sender needs the Merkle tree state to construct the proof. Strategy:

- **Wallet syncs** the Merkle frontier, the user's own note paths, and the demurrage epoch log whenever online
- **Stale root tolerance**: chain accepts any of the last N=100 roots, giving a window of ~100 blocks (~10 minutes at 6s blocks) of staleness
- **Incremental sync**: the Merkle tree is append-only, so the wallet only needs to update the path for new leaves added since last sync
- **Epoch log sync**: the log is tiny (one entry per rate change) and changes at most every few months ‚Äî essentially free to cache

### Shielded Pool Balance Accounting

A dedicated pool account (via `PalletId`) holds all shielded funds:
- `deposit()` transfers from user ‚Üí pool
- `withdraw()` transfers from pool ‚Üí user (after applying `F(current_block)` to reference_value)
- Private transfers only rearrange notes within the pool ‚Äî no on-chain balance movement needed (the pool total stays the same)
- Pool solvency is guaranteed by the value conservation constraint in the circuit plus the reference-value normalization

### Migration Path from Current Design

1. **Phase 0** (current): Keep existing non-private extrinsics working
2. **Phase 1**: Add `deposit`, `submit_private_payment`, `withdraw` alongside existing calls. Add `DemurrageEpochLog` storage and hook into `update_demurrage()` in the communities pallet.
3. **Phase 2**: Deprecate `register_offline_identity` + `submit_offline_payment` (the non-private versions)
4. **Phase 3**: Remove legacy code

Both the old VK (simple circuit) and new VK (private circuit) coexist during transition. Separate storage items, separate call indices.

### Implementation Steps

#### Step 1: Demurrage Epoch Log
- Add `DemurrageEpoch` type and `DemurrageEpochLog` storage (per community)
- Helper: `cumulative_factor(cid, block) ‚Üí BalanceType` ‚Äî reads last epoch entry, extends with current rate
- Hook into `pallet_encointer_communities::update_demurrage()` to append new epoch entries
- Initialize log on first deposit (or community creation)
- Unit tests: factor computation across rate changes, boundary conditions

#### Step 2: Incremental Merkle Tree (`offline-payment/core/src/merkle.rs`)
- Implement fixed-depth (20) incremental Poseidon Merkle tree
- Frontier-based storage (one hash per level = 20 √ó 32 = 640 bytes)
- Functions: `insert(leaf) ‚Üí (root, index)`, `get_path(index) ‚Üí [sibling; 20]`, `root() ‚Üí hash`
- Constraint gadget: `MerklePathVar` for in-circuit verification
- Unit tests for insert, path generation, root computation

#### Step 3: Private Payment Circuit (`offline-payment/core/src/private_circuit.rs`)
- `PrivatePaymentCircuit` implementing `ConstraintSynthesizer<Fr>`
- All 7 constraint groups above (all in reference-value space, no demurrage logic)
- Range check gadget for value non-negativity (binary decomposition, ~64-128 bits)
- Unit tests: satisfied with honest witness, unsatisfied with tampered values (each constraint)

#### Step 4: Pallet Storage & Deposit (`offline-payment/src/lib.rs`)
- Add storage: `MerkleTreeFrontier`, `RecentRoots`, `NextLeafIndex`, `DemurrageEpochLog`

#### Step 5: Private Payment Settlement (`offline-payment/src/lib.rs`)
- Implement `submit_private_payment()` extrinsic

#### Step 6: Withdrawal Circuit & Extrinsic
- Simpler circuit variant (Merkle membership + nullifier, reference_value is public)
- `withdraw()` extrinsic with `payout = reference_amount √ó F(current_block)` conversion

#### Step 7: QR Encoding & Note Encryption (`offline-payment/core/src/note.rs`)
- Note struct with serialization (includes reference_value + UX metadata)
- ECIES encryption/decryption for recipient note data
- QR payload encoding/decoding

#### Step 8: Trusted Setup
- New ceremony for the private circuit (different constraint count ‚Üí different SRS)
- Reuse existing `ceremony.rs` MPC infrastructure

#### Step 9: Benchmarks & Weights
- Benchmark all new extrinsics
- The Groth16 verification cost is similar (constant ~6 pairings), but Merkle tree appends and epoch log reads add storage I/O

### Privacy Properties

| Property | Current Design | New Design |
|----------|---------------|------------|
| Sender identity | Public (AccountId in call) | Hidden (anonymity set = all depositors) |
| Recipient identity | Public (in proof) | Hidden until withdrawal |
| Amount | Public (in proof) | Hidden (only conservation is proven) |
| Payment graph | Fully visible | Unlinkable (deposit ‚Üí transfer ‚Üí withdraw) |
| Nullifier linkability | Links to sender's commitment | Links to nothing identifiable |
| Deposit timing | N/A | Public (but doesn't leak into note values due to normalization) |

### Security Considerations

- **Merkle tree depth**: 20 levels is standard (Tornado Cash uses 20). Supports 2^20 ‚âà 1M notes.
- **Trusted setup**: New circuit requires a new ceremony. The existing MPC infrastructure in `ceremony.rs` applies.
- **Front-running**: A relayer could see a proof in the mempool and front-run submission. Mitigated by the fact that the proof is valid regardless of who submits it ‚Äî the notes are created either way.
- **Griefing**: Someone could submit an invalid proof to waste block space. Mitigated by weight fees.
- **Note encryption**: If the ephemeral key is compromised, the recipient's note details leak. Use fresh randomness per payment.
- **Merkle root staleness**: If the sender's cached root expires (>100 new leaves added while offline), they must resync. For low-throughput community currencies this is unlikely to be a problem.
- **Value overflow**: Range checks in the circuit prevent creating notes with negative values (which would be "free money" in the field arithmetic).
- **Demurrage accuracy**: Offline sender's F(t) estimate error is bounded by block estimation error √ó rate. Even extreme scenarios (1000 blocks off, rate doubled while offline) produce < 0.01% error ‚Äî well within the 0.1% tolerance. See analysis in Demurrage section.
- **Demurrage rate change during offline**: The epoch log captures rate changes precisely. Only the sender's *local estimate* of F(t) is approximate. The on-chain deposit/withdraw conversions always use the exact epoch log. So the only affected value is the split between recipient and change notes ‚Äî the total reference_value in the system is always exactly conserved.

### What This Does NOT Provide

- **Recipient privacy during settlement**: When `submit_private_payment` is called, the two new commitments are public. The recipient knows which one is theirs, but observers don't ‚Äî unless the recipient immediately withdraws.
- **Multi-input transactions**: This design spends exactly 1 note and creates 2. For larger payments, the sender must first merge notes (separate merge transaction) or use multiple transfers. Could be extended to 2-in-2-out later.
- **Cross-community private swaps**: Out of scope. Each payment is within one community currency or native token.
- **Protection against rate changes > 0.1% per offline window**: By design, rate changes are infrequent and small. If governance makes a drastic rate change, offline senders may experience > 0.1% error in the recipient/change split. This is acceptable because: (a) the total value is conserved, (b) drastic rate changes are governance decisions visible to all, (c) the error only affects the single payment in flight.
