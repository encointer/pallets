# Private Offline Payments — Design Plan

## Motivation

The current offline-payment pallet uses ZK proofs but reveals the sender on-chain (the sender's `AccountId` is an explicit parameter, looked up in `OfflineIdentities`). This makes ZK unnecessary — a simple signature achieves the same. To justify ZK, we need **sender anonymity**: hide *which* registered identity is paying, within an anonymity set of all depositors.

## Architecture: Shielded Note Pool

Move from account-based debits to a UTXO/note model. Users deposit funds into a shared shielded pool. Payments spend notes and create new ones without revealing which note was spent.

### Note Structure

```
note = (value, secret, blinding)
note_commitment = Poseidon(value, secret, blinding)
nullifier = Poseidon(secret, leaf_index)
```

- `value`: amount (Fr encoding of BalanceType or u128)
- `secret`: per-note random secret (Fr), generated by the note creator
- `blinding`: random blinding factor (Fr) for commitment uniqueness
- `leaf_index`: position in the on-chain Merkle tree (prevents nullifier linkability across notes)

### On-Chain Storage

1. **Incremental Merkle Tree** (new storage)
   - Append-only tree of `note_commitment` values
   - Fixed depth = 20 (supports ~1M notes, sufficient for community currencies)
   - Poseidon hash for internal nodes (SNARK-friendly, already in the codebase)
   - Store: `leaves: Vec<[u8;32]>`, `next_index: u32`, precomputed subtree hashes for efficient appends
   - Alternatively, store only the frontier (one hash per level) for O(depth) append cost

2. **Recent Merkle Roots** (new storage)
   - Ring buffer of last N roots (e.g., N=100)
   - Allows proofs built against slightly stale state (critical for offline use — sender caches their Merkle path before going offline)
   - On each new leaf insertion, push the new root

3. **Used Nullifiers** (existing, reused as-is)
   - `Map<[u8; 32] → ()>` — shared across all payment types

4. **Remove**: `OfflineIdentities` map — no longer needed (no account↔commitment link)

### New Circuit: `PrivatePaymentCircuit`

**Public inputs (6 field elements):**

| # | Field | Purpose |
|---|-------|---------|
| 1 | `merkle_root` | Which tree state the proof is against |
| 2 | `nullifier` | Double-spend prevention |
| 3 | `out_commitment_1` | Recipient's new note commitment |
| 4 | `out_commitment_2` | Sender's change note commitment (or zero-value note) |
| 5 | `asset_hash` | Community + chain binding (`blake2(cid ∥ genesis_hash)`) |
| 6 | `fee_or_metadata` | Reserved (e.g., for relayer fees or future extensions) |

**Private inputs:**

| Field | Purpose |
|-------|---------|
| `in_value` | Value of the spent note |
| `in_secret` | Secret of the spent note |
| `in_blinding` | Blinding of the spent note |
| `in_leaf_index` | Position of the spent note in the tree |
| `merkle_path[20]` | Sibling hashes along the path to root |
| `out_value_1` | Value going to recipient |
| `out_secret_1` | Recipient's note secret |
| `out_blinding_1` | Recipient's note blinding |
| `out_value_2` | Change value back to sender |
| `out_secret_2` | Sender's change note secret |
| `out_blinding_2` | Sender's change note blinding |

**R1CS constraints:**

1. **Input note reconstruction**: `in_commitment = Poseidon(in_value, in_secret, in_blinding)`
2. **Merkle membership**: Verify `in_commitment` at `in_leaf_index` with `merkle_path` hashes up to `merkle_root` (20 Poseidon hashes)
3. **Nullifier correctness**: `nullifier = Poseidon(in_secret, in_leaf_index)`
4. **Output note 1**: `out_commitment_1 = Poseidon(out_value_1, out_secret_1, out_blinding_1)`
5. **Output note 2**: `out_commitment_2 = Poseidon(out_value_2, out_secret_2, out_blinding_2)`
6. **Value conservation**: `in_value = out_value_1 + out_value_2`
7. **Non-negative outputs**: Range checks on `out_value_1` and `out_value_2` (e.g., check fits in 64 bits — ~128 constraints each using binary decomposition)

**Estimated constraint count**: ~6,000–8,000 (very manageable for Groth16; proof generation ~2-5s on a phone).

### Extrinsics

#### 1. `deposit(amount, cid, note_commitment)` — Shield funds

- Caller pays from their CC or native balance (public, no ZK needed)
- Chain debits caller, appends `note_commitment` to Merkle tree
- Emits `Deposited { who, cid, amount, note_commitment, leaf_index }`
- The caller privately records `(value, secret, blinding, leaf_index)` — needed to spend later

#### 2. `submit_private_payment(proof, merkle_root, nullifier, out_commitment_1, out_commitment_2, asset_hash)` — Settle a private transfer

- Callable by **anyone** (relayer, recipient, sender when back online)
- No sender identity in the call at all
- Steps:
  1. Check `merkle_root` is in the recent-roots ring buffer
  2. Check `nullifier` not already used
  3. Verify Groth16 proof against public inputs
  4. Append `out_commitment_1` and `out_commitment_2` to the Merkle tree
  5. Mark nullifier as used
  6. Emit `PrivatePaymentSettled { nullifier, out_commitment_1, out_commitment_2 }`

- **Note**: No balance check possible here (we don't know who the sender is!). Value conservation is enforced entirely by the circuit.

#### 3. `withdraw(proof, merkle_root, nullifier, recipient, amount, asset_hash)` — Unshield funds

- Proves ownership of a note and reveals the value
- Uses a simpler circuit variant (Merkle membership + nullifier, value is public)
- Chain credits `recipient` with `amount` from the shielded pool
- The shielded pool is a module-controlled account (PalletId-based pot)

#### 4. `set_verification_key_private(vk)` — Governance sets the VK for the new circuit

- Separate VK from the legacy circuit (different constraint system)
- Could store both VKs if backward compatibility needed during migration

### QR Code Payload (Offline Transfer)

The sender constructs the proof offline and encodes everything the recipient needs:

```
Field                           Bytes
─────────────────────────────────────
groth16_proof                     192
merkle_root                        32
nullifier                          32
out_commitment_1 (recipient)       32
out_commitment_2 (change)          32
asset_hash                         32
encrypted_note (for recipient)   ~130
  ├─ ephemeral_pubkey               32
  ├─ ciphertext                     80  (value + secret + blinding + leaf_index_hint)
  └─ MAC                            16
─────────────────────────────────────
Total                            ~482 bytes
```

This fits comfortably in a single QR code (QR Version 15 at medium ECC handles ~520 bytes of binary data).

### Encrypted Note for Recipient

The recipient needs `(out_value_1, out_secret_1, out_blinding_1)` to later spend their note. Encryption scheme:

1. Sender generates ephemeral x25519 keypair `(eph_sk, eph_pk)`
2. Derives shared secret via ECDH: `shared = x25519(eph_sk, recipient_x25519_pk)`
3. Derives encryption key: `enc_key = blake2(shared ∥ eph_pk ∥ recipient_pk)`
4. Encrypts `(value, secret, blinding)` with ChaCha20-Poly1305 or XSalsa20-Poly1305
5. Includes `eph_pk` in the QR so recipient can reconstruct `shared`

**How recipient gets their x25519 key**: Encointer accounts use sr25519. The recipient's x25519 public key can be derived from their sr25519 key (same underlying curve25519 point), or displayed as a separate QR / NFC tag at point-of-sale.

**Two-round in-person flow**:
1. Recipient displays their receiving key (QR at point of sale, NFC tag, etc.)
2. Sender scans it, constructs proof offline, displays payment QR
3. Recipient scans payment QR → submits on-chain when connectivity is available

### Offline Merkle Path Caching

The sender needs the Merkle tree state to construct the proof. Strategy:

- **Wallet syncs** the Merkle frontier and the user's own note paths whenever online
- **Stale root tolerance**: chain accepts any of the last N=100 roots, giving a window of ~100 blocks (~10 minutes at 6s blocks) of staleness
- **Incremental sync**: the Merkle tree is append-only, so the wallet only needs to update the path for new leaves added since last sync
- **Pre-computation**: the wallet can pre-compute partial proofs or cache the proving key

### Shielded Pool Balance Accounting

A dedicated pool account (via `PalletId`) holds all shielded funds:
- `deposit()` transfers from user → pool
- `withdraw()` transfers from pool → user
- Private transfers only rearrange notes within the pool — no on-chain balance movement needed (the pool total stays the same)
- Pool solvency is guaranteed by the value conservation constraint in the circuit

### Migration Path from Current Design

1. **Phase 0** (current): Keep existing non-private extrinsics working
2. **Phase 1**: Add `deposit`, `submit_private_payment`, `withdraw` alongside existing calls
3. **Phase 2**: Deprecate `register_offline_identity` + `submit_offline_payment` (the non-private versions)
4. **Phase 3**: Remove legacy code

Both the old VK (simple circuit) and new VK (private circuit) coexist during transition. Separate storage items, separate call indices.

### Implementation Steps

#### Step 1: Incremental Merkle Tree (`offline-payment/core/src/merkle.rs`)
- Implement fixed-depth (20) incremental Poseidon Merkle tree
- Frontier-based storage (one hash per level = 20 × 32 = 640 bytes)
- Functions: `insert(leaf) → (root, index)`, `get_path(index) → [sibling; 20]`, `root() → hash`
- Constraint gadget: `MerklePathVar` for in-circuit verification
- Unit tests for insert, path generation, root computation

#### Step 2: Private Payment Circuit (`offline-payment/core/src/private_circuit.rs`)
- `PrivatePaymentCircuit` implementing `ConstraintSynthesizer<Fr>`
- All 7 constraint groups above
- Range check gadget for value non-negativity (binary decomposition, ~64-128 bits)
- Unit tests: satisfied with honest witness, unsatisfied with tampered values (each constraint)

#### Step 3: Pallet Storage & Deposit (`offline-payment/src/lib.rs`)
- Add storage: `MerkleTreeFrontier`, `MerkleTreeLeaves` (or events-only), `RecentRoots`, `NextLeafIndex`, `ShieldedPoolId`
- Implement `deposit()` extrinsic (CC and native variants)
- Genesis config for pool account

#### Step 4: Private Payment Settlement (`offline-payment/src/lib.rs`)
- Implement `submit_private_payment()` extrinsic
- New verifier path for the private circuit's VK
- Merkle tree append for output commitments

#### Step 5: Withdrawal Circuit & Extrinsic
- Simpler circuit variant (Merkle membership + nullifier, public value)
- `withdraw()` extrinsic

#### Step 6: QR Encoding & Note Encryption (`offline-payment/core/src/note.rs`)
- Note struct with serialization
- ECIES encryption/decryption for recipient note data
- QR payload encoding/decoding

#### Step 7: Trusted Setup
- New ceremony for the private circuit (different constraint count → different SRS)
- Reuse existing `ceremony.rs` MPC infrastructure

#### Step 8: Benchmarks & Weights
- Benchmark all new extrinsics
- The Groth16 verification cost is similar (constant ~6 pairings), but Merkle tree appends add storage writes

### Privacy Properties

| Property | Current Design | New Design |
|----------|---------------|------------|
| Sender identity | Public (AccountId in call) | Hidden (anonymity set = all depositors) |
| Recipient identity | Public (in proof) | Hidden until withdrawal |
| Amount | Public (in proof) | Hidden (only conservation is proven) |
| Payment graph | Fully visible | Unlinkable (deposit → transfer → withdraw) |
| Nullifier linkability | Links to sender's commitment | Links to nothing identifiable |

### Security Considerations

- **Merkle tree depth**: 20 levels is standard (Tornado Cash uses 20). Supports 2^20 ≈ 1M notes.
- **Trusted setup**: New circuit requires a new ceremony. The existing MPC infrastructure in `ceremony.rs` applies.
- **Front-running**: A relayer could see a proof in the mempool and front-run submission. Mitigated by the fact that the proof is valid regardless of who submits it — the notes are created either way.
- **Griefing**: Someone could submit an invalid proof to waste block space. Mitigated by weight fees.
- **Note encryption**: If the ephemeral key is compromised, the recipient's note details leak. Use fresh randomness per payment.
- **Merkle root staleness**: If the sender's cached root expires (>100 new leaves added while offline), they must resync. For low-throughput community currencies this is unlikely to be a problem.
- **Value overflow**: Range checks in the circuit prevent creating notes with negative values (which would be "free money" in the field arithmetic).

### What This Does NOT Provide

- **Recipient privacy during settlement**: When `submit_private_payment` is called, the two new commitments are public. The recipient knows which one is theirs, but observers don't — unless the recipient immediately withdraws.
- **Multi-input transactions**: This design spends exactly 1 note and creates 2. For larger payments, the sender must first merge notes (separate merge transaction) or use multiple transfers. Could be extended to 2-in-2-out later.
- **Cross-community private swaps**: Out of scope. Each payment is within one community currency or native token.
